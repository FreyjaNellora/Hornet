## Where I diverge from both reviews

### 1. `count_defenders` is outcome-affecting, not "ordering-only"

Fable filed it under "only affects move ordering, not legality." That undersells it, and it changes the priority math. Hornet's search is a **Max^n beam** — depth is bought purely by beam narrowing. In a beam, ordering *is* selection: the top-k ordered moves are the only ones expanded. A backwards free-capture heuristic therefore doesn't merely waste cycles — it can promote a defended (losing) capture into the beam and prune a genuinely free one out of it, **changing the move actually played**.

This is the same root cause Fable already noted and didn't carry forward: the TT is ordering-only *because the beam makes values inexact*. When the beam makes ordering decide the result, an ordering bug is a result bug. So I'd reclassify the defender bug from "cleanup" to "perturbs the data-generating process" — which is why Kimi's independent **P0** (not P2) reads correct to me, though I don't have her stated rationale to confirm she got there the same way.

### 2. The ablation discipline has a hole, and this is what fell through it

Eval weights get bootstrap CIs and default-off arms. Move ordering got `const = true` with no measurement, justified by "ordering doesn't affect correctness." True for legality, false for a beam's *results*. The rigor keeping `(6,0,0,1)` honest isn't applied to the layer that decides what the beam sees. Recommend a Hard Rule amendment: **anything that changes the played move — ordering, beam width, LMR — is under the same default-off + self-play gate as eval**, not a separate lax track.

### 3. Sequencing: the defender fix is the one item that is *not* self-contained

Kimi's plan bundles all P0/P1 into "~70 min of self-contained cleanup." Nine items genuinely are. `count_defenders` is not: fixing the polarity changes ordering → changes the beam → changes played moves → changes self-play outcomes. It can't ride in the same silent batch as doc-comment fixes. Split it:

- **Now, zero-risk:** flip both move-order flags to default-**off**. This is *corrective, not contaminating* — the current default-on-but-buggy state is already off the documented baseline, so reverting to off restores the baseline the other measurements assume. Stops shipping a backwards heuristic immediately; needs no re-validation because it returns to the Hard-Rule-#6 baseline.
- **Later, gated:** fix the polarity and reintroduce the free-capture bonus as a real default-off ablation arm with a measured self-play delta. Turn it on only if it earns its place.

### 4. "Data, not ideas" is not independent of search correctness

Fable's strategic read — the blocker is decisive-outcome data, not ideas — is right but framed as if the two are orthogonal. They aren't. The buggy default-on ordering means a fraction of the existing self-play corpus was generated by a degraded search, so part of your labels are off-baseline. "Get more data" and "fix the search that generates the data" are not parallel tracks; the second gates the first. Concretely: don't spend compute on a larger self-play run *before* flipping those flags off, or you scale up noisy labels. Everything generated after the flip is clean.

## On the blind-review methodology (one correction)

It's tempting to read "Kimi also flagged `count_defenders` P0" as confirmation of Fable's thesis that blind code review catches what doc-aware review misses. It isn't — Kimi **graded Fable's paper**, she didn't retake the exam. Her doc is structured as a scored response to his list, so her agreement is confirmation-of-findings, not an independent blind pass.

The actual evidence for the thesis is upstream and stronger: the *earlier* convergent doc-aware analyses (Claude + Kimi, the ones that diagnosed the eval objective) both ran over this same code and **neither surfaced the stale test assertion or the inverted heuristic**. Those defects survived multiple doc-aware reviews until a cold code pass found them. Kimi confirming them now is consistent with "real and previously missed," which *supports* the thesis — just not for the reason it first appears. If you want to actually test it rather than infer it, the experiment is: have a reviewer go through the code cold, no findings list in hand, and see what class of defect they return.

Net: I endorse adding periodic blind code-only passes to the audit checklist (Kimi's recommendation). The value isn't redundancy with the doc-aware reviews — it's that it sees a *different defect class*: state that lives only in code and never made it into a doc to be reasoned about.
